# 영업 파이프라인 ↔ 관리 웹페이지 연동 스펙

---

## 1. 개요

서비스데스크 영업 파이프라인과 관리 웹페이지가 딜 데이터를 양방향으로 주고받는 연동입니다.

서비스데스크가 push한 데이터를 화면에 표시하고, 관리자가 수정하면 서비스데스크에 신호를 보내 **DB에 저장**합니다.

### 데이터 흐름

```
서비스데스크 DB                     관리 웹페이지
(sales-api)                        (관리 웹 백엔드)

딜 생성/수정/이동/삭제 ──→ POST /deals/web-push ──→  화면에 표시/삭제
        ↑                                                    │
        ├────── POST /deals        ←──  관리자가 딜 신규 생성
        ├────── DELETE /deals/{dealId} ←──  관리자가 딜 삭제
        ├────── POST /deals/dbupdate ←──  관리자가 필드 수정 (서비스데스크 DB에 저장)
        ├────── POST /deals/send-mail ←── 관리자가 메일 버튼 클릭
        └────── GET  /deals/{dealId}/history ←── 소통 이력 조회
```

### 연동 대상 필드

**서비스데스크 → 관리 웹 (web-push / GET /deals 전체 필드)**


| 분류                | 필드명          | 설명                                          |
| ----------------- | ------------ | ------------------------------------------- |
| 메타                | `mailStatus` | 메일 발송 상태. `미발송` 또는 `발송완료`                   |
|                   | `mailSentAt` | 최초 메일 발송일 (`YYYY-MM-DD`). 미발송이면 `""`        |
| 고객 정보             | `회사명`        | 고객사 이름                                      |
|                   | `담당자`        | 고객사 담당자 이름                                  |
|                   | `연락처`        | 전화번호                                        |
|                   | `이메일`        | 이메일 주소                                      |
|                   | `직원규모`       | 예: `51~100명`                                |
|                   | `유입경로`       | 예: `홈페이지`, `검색광고`                           |
| 문의 내용             | `제목`         | 문의 제목                                       |
|                   | `내용`         | 문의 본문                                       |
|                   | `첨부파일`       | 첨부파일 URL 목록 (배열)                            |
| 영업 필드 (수정 가능, 8개) | `직전소통일`      | `YYYY-MM-DD` 형식. 메일 발송 시 자동 갱신              |
|                   | `안내자`        | 담당자 이름 (쉼표 구분 다중 가능)                        |
|                   | `관심모듈`       | 근태, 급여, PCOFF, 인사/조직, 전자계약, 평가, 교육, 채용, 모바일 |
|                   | `진행`         | 진행, 수주, 실주, 중단, 해지, 보류, 미도입                 |
|                   | `데모생성`       | 미생성, 생성요청, 생성완료, 불필요                        |
|                   | `견적`         | 미발행, 발행요청, 발행완료, 불필요                        |
|                   | `미팅`         | 미정, 일정조율, 예정, 완료, 취소                        |
|                   | `비고`         | 기타 메모                                       |


**관리 웹 → 서비스데스크 (dbupdate 대상, 8개 고정)**

`직전소통일`, `안내자`, `관심모듈`, `진행`, `데모생성`, `견적`, `미팅`, `비고`

## 2. 파이프라인 설정 (서비스데스크 담당자)

서비스데스크 UI → **영업** → 파이프라인 설정 → 해당 파이프라인 수정


| 설정 항목           | 설명                                                    |
| --------------- | ----------------------------------------------------- |
| 관리 웹페이지 연동      | ON/OFF 토글                                             |
| 관리 웹페이지 URL     | 관리 웹 서버 base URL (예: `https://admin.example.com`)     |
| 관리 웹페이지 시크릿 키   | 양측이 공유하는 인증 키 (임의 문자열)                                |
| 발송 메일 계정 (From) | 메일 발송 시 사용할 IMAP 계정 이메일 주소 (예: `noreply@example.com`) |


> 설정 후 딜이 생성·수정·이동될 때마다 자동으로 웹에 push됩니다.
> 또는 RightMenu → **관리 웹페이지** 탭 → **관리 웹페이지에 동기화** 버튼으로 파이프라인 전체 딜을 일괄 push할 수 있습니다.

---

## 3. 인증

모든 API 요청에 `X-ADMIN-SECRET` 헤더가 필요합니다.

```
X-ADMIN-SECRET: <your-admin-secret>
```

헤더가 없거나 값이 다르면 `401 UNAUTHORIZED` 응답.

## 4. 관리 웹페이지에서 **구현해야 할** API

### `POST /deals/web-push`

서비스데스크가 딜 생성·수정·이동·삭제 시 관리 웹 서버로 호출합니다.
**관리 웹 백엔드에서 이 엔드포인트를 구현해야 합니다.**

`event` 필드로 이벤트 종류를 구분합니다.

#### Request

```
POST {adminPageUrl}/deals/web-push
Content-Type: application/json
X-ADMIN-SECRET: <your-admin-secret>
```

**딜 생성·수정·이동 시 (`event: "upsert"`)**

```json
{
  "event": "upsert",
  "dealId": "abc123",
  "mailStatus": "발송완료",
  "mailSentAt": "2026-06-05",
  "deal": {
    "회사명": "ABC테크",
    "담당자": "김민수",
    "연락처": "010-1234-5678",
    "이메일": "minsu.kim@example.com",
    "직원규모": "51~100명",
    "유입경로": "홈페이지",
    "제목": "OKR 솔루션 도입 문의",
    "내용": "성과관리와 OKR 운영을 위한 데모 및 견적을 받고 싶습니다.",
    "첨부파일": ["https://example.com/files/문의서.pdf"],
    "직전소통일": "2026-06-05",
    "안내자": "홍길동",
    "관심모듈": "근태",
    "진행": "진행",
    "데모생성": "생성완료",
    "견적": "발행완료",
    "미팅": "완료",
    "비고": ""
  }
}
```

**딜 삭제 시 (`event: "delete"`)**

```json
{
  "event": "delete",
  "dealId": "abc123"
}
```


| 필드           | 타입     | 설명                                                                  |
| ------------ | ------ | ------------------------------------------------------------------- |
| `event`      | string | `upsert` (생성/수정/이동) 또는 `delete`                                     |
| `dealId`     | string | 서비스데스크 딜 고유 ID                                                      |
| `mailStatus` | string | `upsert` 시만 포함. `미발송` 또는 `발송완료`                                     |
| `mailSentAt` | string | `upsert` 시만 포함. 최초 발송일 (`YYYY-MM-DD`), 미발송이면 `""`                   |
| `deal`       | object | `event: "upsert"` 일 때만 포함. 전체 필드 (string, 첨부파일은 string[], 빈값은 `""`) |


#### Response

`200 OK` 를 반환하면 됩니다. 본문 형식은 자유.

```json
{ "ok": true }
```

#### 구현 권장사항

- `dealId`는 역방향 API 호출 시 사용하므로 상태에 보관
- `event: "delete"` 수신 시 해당 딜을 목록에서 제거
- 시크릿 검증은 서비스데스크가 헤더에 포함해서 보내므로, 웹 서버에서 검증 여부는 선택
- 타임아웃은 **10초 이내**에 응답해야 함 (서비스데스크 쪽 timeout이 10초)

## 5. 관리 웹페이지에서 **호출하는** API

### `POST /deals` — 딜 신규 생성

관리 웹페이지에서 직접 딜을 생성합니다. 생성된 딜은 서비스데스크 DB에 저장되고, 서비스데스크가 `POST /deals/web-push (event: "upsert")`로 관리 웹에 다시 push하여 `dealId`를 포함한 최종 데이터를 동기화합니다.

#### Request

```
POST https://5240help.okrbiz.com/api/sales/deals
Content-Type: application/json
X-ADMIN-SECRET: <your-admin-secret>
```

```json
{
  "회사명": "신규테크",
  "담당자": "홍길동",
  "연락처": "010-0000-1111",
  "이메일": "hong@example.com",
  "직원규모": "11~50명",
  "유입경로": "홈페이지",
  "제목": "급여 모듈 도입 문의",
  "내용": "급여 관련 모듈 견적 요청합니다.",
  "첨부파일": [],
  "직전소통일": "2026-06-16",
  "안내자": "박지현",
  "관심모듈": "급여",
  "진행": "진행",
  "데모생성": "미생성",
  "견적": "미발행",
  "미팅": "미정",
  "비고": ""
}
```

- 고객 정보 필드(`회사명`, `담당자`, `이메일`)는 필수
- 나머지 필드는 선택 (생략 시 빈값 `""` 처리)

#### Response

```json
{
  "success": true,
  "dealId": "xyz789"
}
```


| 응답 코드 | 의미                   |
| ----- | -------------------- |
| `200` | 딜 생성 성공, `dealId` 반환 |
| `400` | 필수 필드 누락             |
| `401` | 시크릿 불일치              |


> 생성 완료 후 약 800ms 내에 서비스데스크가 `web-push (event: "upsert")`로 관리 웹에 콜백합니다.

### `DELETE /deals/{dealId}` — 딜 삭제

관리 웹페이지에서 딜을 삭제합니다. 서비스데스크 DB에서도 삭제되며, 서비스데스크가 `POST /deals/web-push (event: "delete")`로 관리 웹에 삭제 이벤트를 콜백합니다.

#### Request

```
DELETE https://5240help.okrbiz.com/api/sales/deals/{dealId}
X-ADMIN-SECRET: <your-admin-secret>
```

#### Response

```json
{ "success": true }
```


| 응답 코드 | 의미           |
| ----- | ------------ |
| `200` | 딜 삭제 성공      |
| `401` | 시크릿 불일치      |
| `404` | 해당 dealId 없음 |


> 삭제 완료 후 약 800ms 내에 서비스데스크가 `web-push (event: "delete")`로 관리 웹에 콜백합니다.

### `POST /deals/dbupdate` — 딜 필드 수정

관리자가 웹에서 딜 데이터를 수정할 때 서비스데스크 DB에 반영합니다.

#### Request

```
POST https://5240help.okrbiz.com/api/sales/deals/dbupdate
Content-Type: application/json
X-ADMIN-SECRET: <your-admin-secret>
```

```json
{
  "dealId": "abc123",
  "changes": {
    "안내자": "홍길동",
    "진행": "진행",
    "견적": "발행완료"
  }
}
```

- `changes`에는 수정된 필드만 포함 (8개 중 변경된 것만)
- `안내자` 수정 시 담당자 이름으로 자동 매핑 (쉼표 구분 다중 가능)
- `직전소통일` 수정 시 고객 커스텀 필드에 반영

#### 필드별 허용 값


| 필드     | 허용 값                                      |
| ------ | ----------------------------------------- |
| `진행`   | `진행`, `수주`, `실주`, `중단`, `해지`, `보류`, `미도입` |
| `데모생성` | `미생성`, `생성요청`, `생성완료`, `불필요`              |
| `견적`   | `미발행`, `발행요청`, `발행완료`, `불필요`              |
| `미팅`   | `미정`, `일정조율`, `예정`, `완료`, `취소`            |


#### Response

```json
{
  "success": true,
  "updatedFields": ["안내자", "진행", "견적"],
  "deal": {
    "직전소통일": "2026-06-04",
    "안내자": "홍길동",
    "관심모듈": "근태",
    "진행": "진행",
    "데모생성": "생성완료",
    "견적": "발행완료",
    "미팅": "완료",
    "비고": ""
  }
}
```


| 응답 코드 | 의미                       |
| ----- | ------------------------ |
| `200` | 업데이트 성공 (최신 딜 8개 필드 반환)  |
| `400` | `dealId` 또는 `changes` 누락 |
| `401` | 시크릿 불일치                  |
| `404` | 해당 dealId 없음             |

### `POST /deals/send-mail` — 메일 직접 발송

관리자가 웹에서 메일 내용을 작성하여 직접 발송합니다.

#### Request

```
POST https://5240help.okrbiz.com/api/sales/deals/send-mail
Content-Type: application/json
X-ADMIN-SECRET: <your-admin-secret>
```

```json
{
  "dealId": "abc123",
  "to": "minsu.kim@example.com",
  "cc": "manager@example.com",
  "subject": "Re: OKR 솔루션 도입 문의",
  "body": "김민수님 안녕하세요.\n\n문의주신 서비스 관련하여 안내드립니다...",
  "attachments": ["https://example.com/files/견적서.pdf"]
}
```


| 필드            | 타입       | 필수  | 설명                      |
| ------------- | -------- | --- | ----------------------- |
| `dealId`      | string   | ✅   | 서비스데스크 딜 고유 ID          |
| `to`          | string   | ✅   | 받는 사람 이메일               |
| `cc`          | string   | ❌   | 참조 이메일 (없으면 생략 또는 `""`) |
| `subject`     | string   | ✅   | 메일 제목                   |
| `body`        | string   | ✅   | 메일 본문                   |
| `attachments` | string[] | ❌   | 첨부파일 URL 목록 (없으면 `[]`)  |


#### Response

```json
{ "success": true }
```


| 응답 코드 | 의미           |
| ----- | ------------ |
| `200` | 메일 발송 성공     |
| `400` | 필수 필드 누락     |
| `401` | 시크릿 불일치      |
| `404` | 해당 dealId 없음 |


> 발송 성공 시 서비스데스크가 다음 항목을 DB에 자동으로 저장합니다:
>
> - 소통 이력에 발신 기록 추가 (`GET /deals/{dealId}/history`로 조회 가능)
> - `mailStatus` → `발송완료`
> - `mailSentAt` → 발송 당일 날짜 (`YYYY-MM-DD`)
> - `직전소통일` → 발송 당일 날짜로 자동 갱신
> - 발송된 메일의 `messageId`를 내부적으로 저장하여 이후 고객 답장 수신 시 해당 딜과 자동 연결

### `GET /deals/{dealId}/history` — 소통 이력 조회

관리자가 특정 딜의 수·발신 소통 이력을 조회합니다.

#### Request

```
GET https://5240help.okrbiz.com/api/sales/deals/{dealId}/history
X-ADMIN-SECRET: <your-admin-secret>
```

#### Response

```json
{
  "success": true,
  "dealId": "abc123",
  "history": [
    {
      "type": "발신",
      "date": "2026-06-05 09:10",
      "subject": "Re: 데모 계정 요청",
      "body": "데모 계정 안내 및 미팅 가능 일정 전달",
      "attachments": ["https://example.com/files/견적서.pdf"]
    },
    {
      "type": "수신",
      "date": "2026-06-04 15:30",
      "subject": "데모 계정 요청",
      "body": "인사팀 검토용 데모 계정 요청",
      "attachments": []
    }
  ]
}
```


| 필드            | 타입       | 설명                                          |
| ------------- | -------- | ------------------------------------------- |
| `type`        | string   | `수신` 또는 `발신`                               |
| `date`        | string   | 소통 일시 (`YYYY-MM-DD HH:mm` 형식)              |
| `subject`     | string   | 메일 제목 또는 소통 요약                             |
| `body`        | string   | 메일 본문 또는 내용 요약                             |
| `attachments` | string[] | 첨부파일 URL 목록. 발신 이력에만 포함, 없으면 `[]`          |


- 이력은 최신순(내림차순)으로 반환
- 이력이 없으면 `history: []` 반환
- **수신 이력 자동 포함**: 발송한 메일에 대한 고객 답장을 IMAP에서 직접 조회하여 이력에 포함.


| 응답 코드 | 의미           |
| ----- | ------------ |
| `200` | 조회 성공        |
| `401` | 시크릿 불일치      |
| `404` | 해당 dealId 없음 |

## 6. 연동 흐름 요약

### 초기 데이터 적재

1. 서비스데스크 RightMenu → **관리 웹페이지** 탭 → **관리 웹페이지에 동기화** 클릭
2. 파이프라인의 모든 딜이 `POST /deals/web-push` 전송됨
3. 관리 웹페이지는 수신한 데이터를 화면에 표시

### 이후 자동 동기화

- 딜 생성 → 800ms 후 자동 push
- 딜 수정 → 800ms 후 자동 push
- 딜 스테이지 이동 → 800ms 후 자동 push

### 관리자 웹에서 딜 생성

1. 관리자가 웹에서 딜 신규 생성 → `POST /deals` 호출 → 서비스데스크 DB에 딜 생성
2. 서비스데스크가 800ms 내 `web-push (event: "upsert")`로 콜백 → 관리 웹 화면에 `dealId` 포함 최신 데이터 반영

### 관리자 웹에서 딜 삭제

1. 관리자가 웹에서 딜 삭제 → `DELETE /deals/{dealId}` 호출 → 서비스데스크 DB에서 딜 삭제
2. 서비스데스크가 800ms 내 `web-push (event: "delete")`로 콜백 → 관리 웹 목록에서 해당 딜 제거

### 관리자 웹에서 수정

- 관리자가 웹에서 필드 수정 → `POST /deals/dbupdate` 호출 → 서비스데스크 DB 반영

### 관리자 웹에서 메일 발송

- 관리자가 메일 작성 후 발송 → `POST /deals/send-mail` 호출 → 서비스데스크가 발송 처리 및 이력 기록

### 소통 이력 확인

- 관리자가 이력 버튼 클릭 → `GET /deals/{dealId}/history` 호출 → 수·발신 이력 목록 표시

### 고객 답장 수신 자동 반영

1. 관리자가 `POST /deals/send-mail`로 메일 발송 → 서비스데스크가 발송 `messageId` 내부 저장
2. 고객이 해당 메일에 답장 → IMAP이 수신 감지
3. 서비스데스크가 `messageId` 기준으로 딜 자동 매칭 → 소통 이력에 수신 기록 추가 → `POST /deals/web-push (event: "upsert")`로 관리 웹에 자동 알림
4. 관리자가 `GET /deals/{dealId}/history` 호출 시에도 IMAP에서 실시간 조회하여 최신 답장 즉시 반영

## 7. 오류 코드 정리


| 코드               | 설명                                |
| ---------------- | --------------------------------- |

| `DEAL_NOT_FOUND` | 해당 `dealId`의 딜이 존재하지 않음           |
